### <a name="optimize-disk-performance"></a>最佳化磁碟效能  
 請遵循下列指導方針來最佳化磁碟效能的 HYPER-V 虛擬環境中執行的客體作業系統：  

|步驟|參考|  
|----------|---------------|  
|設定 HYPER-V 虛擬機器，使用固定大小的虛擬硬碟 (VHD) 選項使用的虛擬磁碟。 固定大小的 VHD 可提供接近以及的功能，例如叢集支援，以及快照集磁碟支援彈性的實體磁碟的效能。|在 HYPER-V 環境中的磁碟儲存體是可透過虛擬 IDE 控制器或虛擬 SCSI 控制器。 不同於舊版的 Microsoft 虛擬化技術，存取虛擬硬碟時，使用虛擬 IDE 控制器或虛擬 SCSI 控制器之間沒有效能差異。 在 HYPER-V 環境中使用有下列的磁碟儲存體選項：<br /><br /> -   **固定大小磁碟-** 大小固定虛擬硬碟 (VHD) 是其中一個資料區塊上會預先配置實體磁碟，根據在建立時定義的最大磁碟大小。 比方說，如果您建立一個 100 GB 的固定大小 VHD，HYPER-V 會配置所有 100 GB 的資料區塊存放裝置，除了建立新的 VHD 時，所需的 VHD 頁首和頁尾的額外負荷。<br />-   **動態擴充磁碟-** 動態擴充 VHD 是其中一個為其初始的虛擬硬碟包含任何資料區塊。 而是在資料寫入至 VHD，但不超過最大的大小建立 VHD 時所指定，會以動態方式配置空間。 例如，100 GB 的動態擴充磁碟一開始包含只有 VHD 標頭，而且必須少於 2 MB 的實體儲存空間。 新的資料寫入虛擬機器的動態擴充 VHD 時，其他實體的資料區塊會配置中的 VHD 檔案，最多 100 GB 的 2 MB 的增量。<br />-   **差異磁碟-** 差異磁碟是一種特殊的動態擴充的 「 上層 」 VHD 相關聯的 VHD 檔案。 此父子式存放裝置拓撲，在父磁碟會保持不變，「 子 」 差異磁碟只對所做的任何寫入作業。 任何讀取的作業會先檢查對差異磁碟，以查看更新的內容是否已寫入差異磁碟;如果內容不在差異磁碟，從父代 VHD 讀取內容。 差異磁碟是適用於您需要維護特定的基準組態，並想要輕鬆地測試，並接著回復變更為基準的案例。 儘管透過差異磁碟提供的父子式存放裝置拓撲的彈性可用於測試，這不是效能的最佳組態因為沒有與維護的父子式拓樸相關聯的額外負荷使用差異磁碟時才需要。<br />-   **穿透式磁碟 –** 穿通磁碟功能可讓客體作業系統，略過 HYPER-V 主機檔案系統，並直接存取磁碟。 可透過傳遞的客體作業系統的磁碟必須設為 「 離線 」 在 HYPER-V 主機，以確保，在主機和客體作業系統請勿嘗試同時存取磁碟。 傳遞磁碟提供邊際效能會優於其他磁碟儲存體選項，但不支援某些功能的虛擬磁碟，例如虛擬機器快照，以及叢集的支援。 因此 「 穿通磁碟 」 功能不是建議使用在 BizTalk 或 SQL Server 環境中因為臨界效能優勢會超過偏移缺少的功能。<br /><br /> 提供與 HYPER-V 磁碟儲存體選項的相對效能的詳細資訊，請參閱部落格文章 「 HYPER-V 儲存體分析 」，網址[ http://go.microsoft.com/fwlink/?LinkID=132848 ](http://go.microsoft.com/fwlink/?LinkID=132848)。|  
|設定為使用 SCSI 控制站的資料磁碟區的磁碟|這是建議使用，因為如果而模擬的 IDE 控制器是可用，而不需要安裝 HYPER-V 整合服務，已安裝 HYPER-V 整合服務，可以只安裝 SCSI 控制器。 使用 integration services 提供 IDE 篩選器驅動程式執行的磁碟 I/O 是遠優於磁碟 I/O 效能提供模擬的 IDE 控制器。 因此，若要確保最佳的磁碟 I/O 效能，如 HYPER-V 虛擬化環境中的資料檔案，在主機和客體作業系統上安裝 integration services，而且與綜合 SCSI 控制站設定為資料磁碟區的磁碟。 對於跨越多個資料磁碟機的極大量的儲存體 I/O 工作負載，每個 VHD 應該附加至個別的綜合 SCSI 控制器以提升整體效能。 此外，每個 VHD 應該儲存在個別的實體磁碟上。 **重要事項：** 不要附加到 SCSI 控制器的系統磁碟。 包含作業系統的虛擬硬碟必須連接到 IDE 控制器。|  

### <a name="optimize-memory-performance"></a>記憶體效能最佳化  
 請遵循下列指導方針來將 HYPER-V 虛擬環境中執行的客體作業系統的記憶體效能最佳化：  

|步驟|參考|  
|----------|---------------|  
|確保有足夠的記憶體主控 HYPER-V 虛擬機器的實體電腦上安裝|可用的實體記憶體，通常是最重要的效能因素，以便讓 BizTalk Server 為 HYPER-V 虛擬機器上執行。 這是因為每個虛擬機器必須位於非-分頁集區記憶體或無法分頁至磁碟的記憶體。 因為非-分頁集區記憶體無法分頁至磁碟，主控虛擬機器的實體電腦應該有可用的實體記憶體的記憶體總和等於配置給每部虛擬機器以及下列所述：<br />     300 MB 的 Hypervisor 以及 32MB 的第一個 GB 的 RAM 配置給每個虛擬機器加上另一個 8 MB 的每個額外 GB 的 RAM 配置給每個虛擬機器加上主機的 512 MB在根磁碟分割上執行的作業系統<br />     例如，如果 HYPER-V 虛擬機器會配置 2 GB 的記憶體，在 HYPER-V 管理員中，執行 HYPER-V 虛擬機器會大約 2388 MB 時使用的實際實體記憶體 (300 MB 的 hypervisor 的虛擬機器配置 + 2 GB +32 MB + 8 MB = 2388 MB)。 Hypervisor 只需要載入一次，因為初始設定後續的虛擬機器不會產生與載入 hypervisor 相關聯的 300 MB 額外負荷。 因此，如果兩個 HYPER-V 虛擬機器每個配置的 2GB 的記憶體，HYPER-V 管理員中，執行這些 HYPER-V 虛擬機器時所使用的實際實體記憶體會大約 4476 MB （hypervisor 的 300 MB + 4 GB 配置給虛擬機器 + 64 MB + 16 MB = 4476 MB)。 **注意：** 一般的經驗法則，計劃配置至少 512 MB 的記憶體，以便讓根資料分割提供服務，例如 I/O 虛擬化、 快照集檔案支援，以及子資料分割管理。<br />-   **使用 64 位元的客體作業系統盡可能**– 請考慮使用針對每個客體作業系統的 64 位元作業系統。 這應該因為根據預設，32 位元 Windows 作業系統只可以處理最多 2 GB 的每個處理序虛擬位址空間。 安裝在 64 位元作業系統可讓應用程式，以充分利用主控 HYPER-V 虛擬機器的實體電腦上安裝的記憶體。|  

### <a name="optimize-network-performance"></a>最佳化網路效能  
 HYPER-V 虛擬機器中支援綜合和模擬網路介面卡，但是綜合裝置提供大幅提升效能並減少 CPU 額外負荷。 每一張介面卡會連接到虛擬網路交換器，如果需要外部網路連線可以連線至實體網路介面卡。 請遵循本節中的建議，以將 HYPER-V 虛擬環境中執行的客體作業系統的網路效能最佳化。
